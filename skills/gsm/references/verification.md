# GSM verification record

Clean-room protocol, per the FUSION hard rule: fresh `git clone` of the public
repository, fresh build from the unpacked sources, fresh working directory with
**no pre-existing reference `.out`**, stderr captured and inspected, exit status
checked. Nothing below is a `diff` that could pass by accident.

Date: 2026-07-20. Platform: macOS 15 (Darwin 24.6.0), arm64.
Toolchain: Homebrew GCC 15.2.0 (`g++-15`) driven through OpenMPI 5.0.8 `mpic++`,
`-fopenmp -O2 -w -fpermissive`.
Source: `github.com/GSMUTNSR/book_codes` at `ae59a58` ("Bug in GSM-CC and code upgrade"), GSM-2.0.

## Why the comparison is numeric and not a diff

The `Exercises/**/*.out` files shipped with the repository were generated by
**GSM-1.0** (the exercise tree is dated April 2021; the sources are the June 2023
GSM-2.0 upgrade). GSM-2.0 changed the pole-line print format:

```
GSM-1.0 (reference):  proton 0s1/2   k:(0,1.085...) fm^(-1)   E:-24.42... MeV
GSM-2.0 (produced):   proton 0s1/2   k : (0,1.085...) fm^(-1)   E : -24.42... MeV
```

so a byte diff reports differences on every pole line while the physics is
identical. Last-digit rounding also varies with compiler and architecture.
`scripts/compare_gsm.sh` therefore extracts every number and reports relative
differences, split by magnitude: quantities above 1e-2 are the actual
observables (energies in MeV, widths in keV, momenta in fm^-1, norms), while
everything below is dominated by convergence residuals (Jost-function residuals
~1e-13, Berggren completeness overlaps ~1e-8, numerical-zero widths of bound
states) whose relative differences are meaningless.

## Case 1: Chapter 2, Exercise XV A, narrow neutron resonance

Pole search for a 0d3/2 neutron in a Woods-Saxon well (A = 12, V0 = 63 MeV,
Vso = 7.5 MeV, R0 = 3 fm, a = 0.65 fm), with the width also obtained
independently from the current formula and from complex scaling.

```
bash scripts/run_gsm.sh res \
     $GSM_ROOT/Exercises/Chapter_2/Exercise_XV/Exercise_XV_A_neutron_narrow_resonance.in \
```

## Case 2: Chapter 3, Exercise XIII, 1s1/2 proton in a foreign basis

Diagonalization of a Woods-Saxon potential whose Coulomb term uses 8 protons, in
a Berggren basis generated by a different potential (10 protons). This is the
non-trivial test: basis and Hamiltonian deliberately disagree, so the result
depends on genuine completeness of the Berggren set rather than on the basis
already containing the answer. Three overlap conventions (`off`, `sub`, `cut`)
are computed and cross-checked against direct integration.

```
bash scripts/run_gsm.sh one \
     $GSM_ROOT/Exercises/Chapter_3/Exercise_XIII/Exercise_XIII_1s1I2.in ./work
bash scripts/compare_gsm.sh \
     references/Exercise_XIII_1s1I2.out.reference ./work/Exercise_XIII_1s1I2.out
```

Exit status 0, stderr empty. 185 lines produced, 185 in the reference.

contour), MSGI-COSM interaction, giving the full spectrum, densities, and

## Traps caught while producing this record

1. **Exit 139 with empty stderr and no error message.** Caused by the
   `finite()` self-recursion described in SKILL.md, not by the input. The crash
   happens inside a function prologue writing to the stack guard page, so it
   looks like a memory bug in the physics code and is not. Raising the stack
   limit or relinking with a larger `-stack_size` only moves the crash. The fix
   is the `std::isfinite` patch, applied automatically by `install_gsm.sh`.

   The first version of that patch was itself fragile, in two ways worth
   recording because both fail *silently* into the same exit 139:
   its guard matched one exact spelling (`= finite (x);`), so any reformatting
   upstream would skip the patch with no warning; and it did not force a
   rebuild, so a tree compiled before the patch kept its recursing binary,
   since the build step sees the binary already present and skips it.
   The current script tolerates arbitrary spacing, verifies the self-call is
   gone afterwards, aborts if it recognises neither the bug nor the fix, and
   deletes all objects and binaries whenever it patches. Verified on four
   paths: fresh clone, pre-patch build, reformatted source, and re-run.
2. **`make` output polluting a captured variable.** `install_gsm.sh` prints
   `GSM_ROOT=` on stdout and everything else on stderr, because `run_gsm.sh`
   captures stdout. Build chatter on stdout produced a nonsensical binary path.
3. **A path with digits in it read as data.** The many-body output echoes
   `STORAGE_DIR:` with an absolute path; when the workspace lives under a
   directory whose name contains numbers, a naive number extractor counts them
   as results and reports a spurious count mismatch. `compare_gsm.sh` skips
   that line, along with timing lines.
