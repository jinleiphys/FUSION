# KSHELL input format

A KSHELL run has two inputs: an interaction file (`.snt`, shipped in `snt/`) and a
partition file (`.ptn`, generated by `gen_partition.py`), tied together by a
Fortran namelist that `kshell.exe` reads as its single command-line argument.

```
kshell.exe case.input
```

Documented from the shipped `snt/` files, the generated `.ptn`, and the test
scripts (`test/Ne20_usda/Ne20_usda.sh`), not from memory.

## Interaction file `.snt`

The model space and the one- and two-body matrix elements. Header (from
`usda.snt`):

```
! model space
   3   3     8   8          <- n proton orbits, n neutron orbits, proton core, neutron core
    1     0   2   3  -1  !   1 = p 0d_3/2     <- index, n, l, 2j, tz
    2     0   2   5  -1  !   2 = p 0d_5/2
    ...
```

then the single-particle energies and the two-body matrix elements. `usda.snt` /
`usdb.snt` are the Brown-Richter sd-shell interactions (^16O core); `gxpf1a.snt`,
`kb3g.snt`, `fpd6.snt` are fp-shell (^40Ca core); `jun45.snt`, `jj44*.snt` are the
upper fp-g region; `sn100*.snt` the Sn region; and others. `ls snt/` lists all 24
shipped interactions. Pick the one whose model space contains your nucleus.

## Partition file `.ptn`

Generated, not hand-written:

```
gen_partition.py <interaction.snt> <case.ptn> <valence_p> <valence_n> <parity>
```

`valence_p` / `valence_n` are the numbers of valence protons and neutrons ABOVE
the interaction's core, and `parity` is `1` or `-1`. For ^20Ne with USDA (core
^16O, Z=8 N=8): 2 valence protons and 2 valence neutrons, parity 1, so
`gen_partition.py usda.snt Ne20.ptn 2 2 1`. The generator prompts for a truncation
scheme on stdin; feeding `0` selects no truncation (the full space). It writes the
proton and neutron partitions and the allowed combinations:

```
# partition file of usda.snt  Z=2  N=2  parity=1
 2 2 1
 6 6            <- number of proton, neutron partitions
 ...
```

`run_kshell.sh` drives this: pass the interaction and the valence numbers and it
generates the partition, then runs.

## The kshell namelist (`&input ... &end`)

From the 20Ne benchmark:

```
&input
  fn_int = "usda.snt"       ! the interaction (.snt) file
  fn_ptn = "case.ptn"       ! the partition (.ptn) file
  mtot = 0                  ! 2*M of the M-scheme (0 for even-A, 1 for odd-A)
  n_eigen = 5               ! number of lowest states to compute
  hw_type = 2               ! oscillator hw rule (2 = 45 A^-1/3 - 25 A^-2/3), affects transitions not energies
  is_double_j = .false.     ! M-scheme (.false.) vs fixed-J (.true.)
  max_lanc_vec = 200        ! Lanczos vector cap
  maxiter = 300             ! max iterations
  n_restart_vec = 15        ! thick-restart block size
  mode_lv_hdd = 0           ! 0 = keep Lanczos vectors in memory
  eff_charge = 1.5, 0.5     ! (e_p, e_n) effective charges for E2 transitions
  gl = 1.0, 0.0             ! orbital g-factors, for M1
  gs = 5.585, -3.826        ! spin g-factors, for M1
&end
```

Only `fn_int`, `fn_ptn`, `mtot` and `n_eigen` change the eigenvalues; the charges
and g-factors affect only the transition operators (used by `transit`, a separate
executable). `run_kshell.sh` fills a sensible default namelist and exposes the
interaction, valence numbers, parity, `mtot` and `n_eigen` as arguments.

## The interactive UI (not used by the skill)

`bin/kshell_ui.py` is the interactive front end: it asks for the nucleus and
interaction and writes both the partition and a run shell script. The skill
bypasses it and calls `gen_partition.py` plus `kshell.exe` directly for a
reproducible, non-interactive run. The UI (and `gen_partition.py`) are Python 3 in
the maintained GaffaSnobb fork; the older jorgenem mirror is Python 2.
