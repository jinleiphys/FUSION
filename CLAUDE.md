# FUSION: agent rules

One-sentence goal: build FUSION, a rebrand fork of opencode that ships nuclear-physics skills, per-code expert skills, and a self-contained 62k-paper knowledge base; full statement in [README.md#project-goal](README.md#project-goal).

## Hard rules

- **Never modify opencode functional code.** The fork touches brand assets only (logo ASCII art, name strings, icons, splash). Any domain feature goes in the customization layer: skills, agent definitions, MCP servers, opencode config. If a feature seems to need a core change, stop and discuss; that is a red flag, not a task.
- **Keep the brand patch minimal and rebasable.** One commit (or small series) on top of upstream; CI rebases weekly. If rebase conflicts grow, the patch is too big.
- **Skills follow the existing house style.** Reference implementation: `~/.claude/skills/fresco/SKILL.md` (per-code skill) and the other skills in `~/.claude/skills/`. Each per-code skill must include: install/build, input authoring, run, output parsing, and at least one benchmark case with a published reference value.
- **Private-code boundary (HARD RULE, user 2026-07-20).** A per-code skill may cover a code ONLY if that code is BOTH publicly open-source (a public repo anyone can clone) AND has a corresponding published paper. This applies to the user's own codes too: STARS/pstars (internal repo only) and any unpublished code NEVER get a FUSION skill. FUSION is publicly distributed to students and peers, so it may ship only what they can legally obtain. arXiv-preprint-with-public-repo (paper still in journal review) is a per-code judgment call, ask the user.
- **Knowledge base reuses literature-corpus.** Interface and constraints in `~/.claude/skills/literature-corpus/SKILL.md`. Do not build a parallel index; extend query.py / corpus.db (domain-keyword tag layer, MCP wrapper). Lexical BM25 stays the base; no pre-embedding of the full corpus (logged decision, see devlog).
- **Validation uses real cases, not synthetic evals.** Phase 0 gate: 3 skills ported to opencode + a domestic model, each judged on a real task the user has actually done, compared against Claude Code output.
- **No em-dashes** in any generated text, Chinese or English (this file included).
- **Do not ship the user's personal wiki.** ~/research-wiki and ~/research-wiki-personal are private. FUSION only defines the mount point.
- **Corpus redistribution is an open legal question** (arXiv non-exclusive license does not permit public full-text redistribution for most papers). Until resolved (see TODO Phase 3), any public artifact ships metadata + index-without-fulltext or a rebuild-from-arXiv script, never the raw .tex or full-text corpus.db.

- 2026-07-20: **benchmark tiers are explicit and stated honestly per skill.** Tier 1 (FRESCO, COLOSS, CCFULL, GSM, TALYS): the distribution ships reference values and the skill reproduces them to N significant figures. Tier 2 (pikoe and any code whose distribution lacks reference output): the skill still ships, because its goal is INPUT ALIGNMENT, and an input manual plus working sample decks is enough to build it (user ruling). A tier-2 skill is verified as builds + runs + physically sensible + internally consistent across the code's own samples, and must say so plainly rather than implying a numerical benchmark it does not have. Upgrade a tier-2 skill when reference values become available (published figures, or the authors).
- 2026-07-23: **skills/ holds research skills as well as per-code skills.** First one embedded: `skills/exfor-data/` (IAEA EXFOR experimental-data retrieval), which drives no code and so has no build or benchmark step; its integrity check is that parsed values match the raw entry. It is the natural partner to every reaction skill, since comparing a calculation with experiment needs real measured data, and it is deliberately built to report that no measurement exists rather than to stretch a nearby dataset. Also on this date the fresco skill gained `scripts/omp.py`, which emits KD02 and CH89 global nucleon optical potentials as `&POT` blocks with the `ap=0` radius convention applied; pure Python, `--selftest` pins 39 values against the reference Fortran and needs no Fortran toolchain.
- 2026-07-21: **cross-build reproduction, not the author's reference output, is the build-integrity check** (user ruling; amends the 2026-07-20 tier decision). A distributed reference output is produced by the SAME source as your own run, so it can certify only that your build is sound; a genuine physics bug sits in the author's reference too and matching it proves nothing. Reproducing across compilers, architectures and optimization levels certifies the same property over more configurations. Standard sweep: two architectures, two compiler major versions, and `-O2` / `-O0` / `-finit-real=snan -finit-integer=-99999`, with a comparator that REJECTS non-finite values. `-O0` vs `-O2` agreement rules out optimization-sensitive UB; snan agreement rules out uninitialized variables reaching output. Measured on pikoe: bit-identical across all six builds, 5642 numbers. Consequence: **do not email authors for missing reference output**, and physics correctness must be carried separately (published figures or tables). Harness: `skills/pikoe/scripts/xbuild_check.sh` + `xbuild_compare.py` (see devlog 2026-07-21).
- 2026-07-22: **constructibility requires a COMPLETE input set, including signs and phases; a table of results is not by itself an anchor.** This qualifies the 2026-07-21 decision below, which was right for 16O(p,g)17F and wrong as a general rule. AZURE2's 14N(p,g)15O case has a better-looking anchor than 16O (Table IV, S(0) per transition plus a total) and is nonetheless NOT reconstructable: Table II covers "the three strongest transitions" only, the 5.18 MeV final state has neither an ANC nor a gamma width anywhere in the paper, and decisively the SIGNS of the reduced-width amplitudes are unpublished while the ground-state S(0) is set by destructive interference among four 3/2+ components. Picking signs would be inventing the answer. Before promising a constructed benchmark, audit the INPUT side for completeness, not the output side for attractiveness. Tractable subsets can still be worth building (here the 6.79 MeV transition, 72% of the total, external-capture dominated and explicitly "added incoherently" so sign-free)
- 2026-07-21: **a benchmark case may be CONSTRUCTED from published numbers, not only obtained from a distribution** (user observation, on AZURE2). If a paper tabulates a complete parameter set, the input deck is buildable from it and the absence of a shipped example is not a blocker. Prefer a **numeric** anchor (a table of results) over a figure read, because a hand-built deck can encode a format error that produces a wrong but plausible result, and only digits catch that. AZURE2 case: PRC 81, 045805 Sec. IV, Table V (¹⁶O(p,γ)¹⁷F, cleanest), Table I (¹²C+p), Table IV (S factors, total 1.81 keV b, the numeric anchor).
- 2026-07-21: **when a code has both a gated and an open distribution channel, build from the open one and say so.** AZURE2 is GPL-3.0 and anonymously clonable at github.com/rdeboer1/AZURE2 while azure.nd.edu sits behind `login.php`; the skill uses GitHub only, so the private-code boundary is satisfied. A gate on a secondary channel does NOT make a code unobtainable. Contrast GEMINI++, dropped the same day because no public source exists anywhere. Record which channel a skill uses.
- 2026-07-22: **"truly open" means BUILDABLE FROM SOURCE ON THE DISTRIBUTION'S TARGET PLATFORM, not merely that source exists** (user ruling, on GEF). This tightens the private-code boundary. GEF ships its FreeBASIC source (33 `.bas`) and is GPL-3.0 with a published paper, so it passes the boundary as literally written; but FreeBASIC has no native Apple-Silicon toolchain (`fbc` is absent, has no Homebrew formula, and upstream targets x86 only, runnable on ARM only under Rosetta), so a Mac-using student can only run the vendor's precompiled x86/Linux binary and cannot rebuild it. For a project distributed to students who are mostly on Macs, a code you can only run as an opaque vendor binary is not meaningfully open. **Rule: a per-code skill requires source that builds from a toolchain obtainable on the target platform; binary-only-in-practice fails even when a source tarball exists.** GEF dropped on this ground. Note it is a scope test on the DISTRIBUTION, not on physics: running GEF on heliumx is fine for the user's own work, it just cannot ship as a FUSION skill. Consequence for the fusion paper: the fission/statistical row does NOT re-open, because TALYS already carries it at tier 1 (`n-Th232-fis-wkb`, 128/128 bit-identical, plus a fission-yield model), pending the user's call on whether one code may satisfy two subfield rows.
- 2026-07-22: **a deterministic Monte Carlo code with shipped reference output can reach TIER 1, so do not assume MC means "statistical anchor only"** (learned building CGMF). CGMF is Monte Carlo but integer-deterministic (event i is seeded from i), and the repo ships byte-exact `.reference` history files, so the benchmark reproduces the DISTRIBUTED output bit-for-bit rather than settling for nu-bar within some sigma. The earlier assumption (recorded for GEF: "Monte Carlo, so any reference needs the event count pinned and agreement stated as statistical, never bit-identical") is only half right: it holds for a non-deterministic MC, but a deterministic one with a fixed per-event seed is bit-reproducible on a matching build. Before tiering an MC code, check for (i) a fixed/derivable seed and (ii) shipped reference output; if both, aim for tier 1. Keep the two claims separate in the docs: the regression reproduction is exact, the physics observable is still statistical. Applies next to SkyNet and YAHFC.
- 2026-07-22: **a harness guard is not verified until it is shown to FLIP when disabled, and each guard needs a test that ISOLATES it** (process rule, after the CGMF and AZURE2 harnesses each shipped a guard-test that silently proved nothing). Two recurring failure shapes, both are the test lying rather than the code: (i) the disable-edit does not actually apply (bad escaping, wrong line), so the suite stays green and the guard looks tested when it is not, fix by asserting the file changed before running; (ii) the test input trips a DIFFERENT guard first, so disabling the target guard changes nothing, fix by making each negative case fail ONLY the guard under test (e.g. a wrong-Einc stub must still be full-length, or the fragment-count guard rejects it instead). Standing procedure for any new guard: write a negative case, confirm the suite is green, disable the guard, confirm exactly that one case flips, restore. This sits alongside the wrong-operand rm -rf lessons in [[devlog-archive]]: writing the guard is not the work, proving the guard fires is.
- 2026-07-22: **check the README and the paper's program summary for a license declaration, not just a LICENSE file or GitHub's detector; both miss a license stated in prose.** On KSHELL (Shimizu et al., CPC 244, 372 (2019); arXiv:1310.5431) the initial read was "no license" because there is no LICENSE file and GitHub reports license:None for both the jorgenem mirror and the maintained GaffaSnobb fork. The user's call was to proceed anyway (public-clonable + published, and the skill clones from upstream rather than redistributing). The Codex pass then found the upstream README's CPC-style header states **"Licensing provisions: GPLv3"**, so KSHELL IS GPL-3.0 by the authors' declaration, just missing the standalone LICENSE file GitHub's detector keys on. So it passes the private-code-boundary "publicly open-source" clause cleanly, and the near-miss was a detection failure on my side (trusting the LICENSE-file/GitHub signal over the README prose), not a genuine boundary edge. Lesson for future codes: grep the README and the paper for "licen"/"GPL"/"copyright" before concluding a code is unlicensed. Separately, and still true: use the maintained GaffaSnobb fork (Python 3.10+ tooling), not the unmaintained jorgenem mirror (Python 2 gen_partition/kshell_ui that will not run on a modern Mac).

- 2026-07-22: **a "reproduce the shipped reference" benchmark that filters out per-run noise (timestamps, cpu-time) must anchor the filter to the EXACT noise lines, never a broad token match, or the filter hides real differences on both sides.** SWANLOP writes a timestamp header into every `zz.*` file, so verify strips it before diffing against the shipped `.REF`; the first cut, `grep -viE 'Date:|Time:|UTC|CPU'`, removed ANY line containing those tokens from both the fresh output and (implicitly) matched a bogus appended `CPU 999 WRONG DATA` line out of existence, so a corrupted output diffed clean. The fix anchors to the two real timestamp forms only (`Date:.*Time:` and `^# NNNN:NN:NN ... UTC`), and drops `CPU` entirely once the `.REF` were confirmed to carry no cpu line. Rule: when normalizing output before a reference diff, match the noise pattern as tightly as the noise actually is (anchored, comment-only, specific), and verify a deliberately corrupted line still survives the strip and fails the diff. This is the diff-normalization cousin of the guard-must-flip rule.

- 2026-07-22: **a code that ships no reference output is still a legitimate tier-2 skill when cross-build agreement plus a code-internal physics identity pin it.** SIDES ships an input manual and a runnable example but no reference numbers, and its integro-differential iteration is not bit-identical across gfortran versions (macOS 15.2 vs Linux 13.3 agree to ~12 sig figs, ~1e-11, not bit-for-bit, unlike a non-iterative code such as CNOK/pikoe). Verified by: a regression gate against the pinned value (1e-9 relative, two orders over the observed cross-version drift and far under a real bug), plus the neutron optical theorem TOTAL=ELASTIC+REACTION holding to machine precision (an internal identity that catches elastic/reaction being independently wrong). State the gate as a regression check, not a re-run of the cross-build, and record the cross-build agreement as documented evidence. Proton runs carry no such identity (Coulomb makes the integrated elastic/total ill-defined and SIDES writes reaction only), so gate them differently, do not force a four-column neutron parse onto a two-column proton file.

- 2026-07-22: **when a build needs a source patch to compile on the target platform, prove the patch is behaviour-preserving by CROSS-BUILD BIT-IDENTITY, not by argument.** The decisive test is a three-corner match: the code built UNPATCHED on its native platform (where it compiles) must equal the code built PATCHED on the target platform, and equal the code built PATCHED on the native platform, all bit-for-bit on the same numeric output. If those agree, the patch changed portability and not physics; if they diverge, the patch is a physics change wearing a portability costume. Measured on CNOK: `fabs(std::complex)` compiles on gcc/libstdc++ (returns the magnitude) but not on Apple clang/libc++; rewriting it to `std::abs` gave output bit-identical to the gcc-unpatched build across 60.086689/18.056073/78.142761 mb (macOS clang patched -O2 and -O0; Linux gcc unpatched; Linux gcc patched). This is stronger than the 2026-07-21 cross-build rule (which certifies the build) because it also certifies a HAND EDIT to the source. Corollary: a numeric anchor documented in the PAPER can drift from the released code (CNOK's paper diffractive/total sit 0.03% off the current commit); cross-build bit-identity localizes such a residual to the paper side rather than the build, so do not chase it as a build error.

- 2026-07-23: **when a code is NOT bit-reproducible across platforms, a portability patch is proven behaviour-preserving by CROSS-BUILD REPRODUCTION OF THE SHIPPED REFERENCES, not bit-identity; and the honest tier is "tier 1 with a documented, narrowly-encoded platform caveat".** This qualifies the 2026-07-22 CNOK bit-identity rule, which assumed a non-iterative code where two builds agree bit-for-bit. SkyNet integrates stiff ODEs and solves nonlinear NSE (Saha) systems, so architecture/libm differences produce last-figure (and, for the T9=3 full-network NSE where abundances span ~200 decades, larger) differences; bit-identity is unattainable. The patch is instead certified by building the SAME patched source on the other platform and passing the code's OWN shipped self-comparison suite there: SkyNet's CTest is 19/19 on Linux (glibc) and 17/19 on macOS. The two macOS misses were localized, not waved away: a wall-clock timing self-test (StopWatch, environment-flaky), and the single stiffest NSE reference (7.0e-3 vs a glibc-calibrated 3.5e-5 gate). The NSE miss is a genuine Apple-libm vs glibc `exp`/`log` difference in a stiff solve, PROVEN by (i) identical output under `-O3` and `-ffp-contract=off` (so not FMA contraction or optimization UB) and (ii) the identical source passing on Linux (so not the patch). Consequences: (a) do not chase a libm-limited stiff-solve delta with compiler flags or by loosening the passing platform's gate; document it. (b) encode the platform exception NARROWLY in verify: the other blocks/tests must still pass on BOTH platforms (a real regression breaks them), and the excepted case is allowed only within a bounded WINDOW around its measured value, never merely "small". (c) state the tier as tier-1-with-caveat, not a bare tier-1 or a downgrade to tier-2. Cross-build here needs the deps on the other box; conda-forge (`hdf5 gsl boost cmake`) supplies them on heliumx without sudo.

- 2026-07-23: **a registered-package skill (Julia Pkg, and by extension pip/conda) installs into an ISOLATED, VERSION-PINNED environment and benchmarks by reproducing the package's own shipped test suite; it needs no source patches.** Established on NuclearToolkit.jl, the first Julia-package skill. Pattern: (i) `Pkg.add(name=..., version=PIN)` into a dedicated `JULIA_DEPOT_PATH` + project under the fusion cache, so the user's global environment is never touched or upgraded, and set that depot on EVERY interpreter call (install, run, verify, and any stub), or a stray call silently loads a different version from the user's home. (ii) No cross-build / bit-identity question: the language toolchain handles portability and the pin fixes the numerics, so tier-1 evidence is reproducing the package's `@test` (or pytest) references, via a fast self-contained numeric anchor plus the full `Pkg.test`. (iii) Guard the same false-success modes the Codex pass caught here: verify/run must require the interpreter's EXIT CODE == 0 in addition to parsing content (a crash after printing partial output otherwise passes); the full-suite check must require the EXACT pinned test count (accepting "total >= N" lets a skipped/partial suite falsely certify); the fast path must compare the ACTUAL installed version to the pin (an altered manifest can drift the version while the stamp and a physics probe still pass); and any version/override string interpolated into interpreter source must be validated (semver) or passed as an argument, not spliced into code. Confirmed-clean and reusable: passing values to the interpreter as ARGS rather than interpolating them is injection-safe. Contrast the compiled skills (CNOK/SkyNet), where portability patches and cross-build reproduction are the core work; for a package skill that work simply does not exist.

- 2026-07-23: **the private-code boundary tests OBTAINABILITY, not the licence's commercial terms** (user ruling, on Sky3D; amends the "publicly open-source" clause of the hard rule). The only question is whether a student can get the source and build it on their own machine. What that student may then do with it commercially is between them and the authors, and the skill's job is to state the terms accurately, not to enforce them. A code therefore passes when the source is freely obtainable AND builds on the target platform, even if its licence restricts use to non-profit. Note for the record that the user's premise, "published means everyone may use it", is not true in general (publication grants no licence, copyright survives it), but the conclusion holds here for a concrete reason: the CPC non-profit use licence DOES grant exactly what FUSION's users, students and academic researchers, need, and the skill clones from upstream and redistributes nothing, so every user receives the code directly from the authors under the authors' own terms, precisely as if they had downloaded it themselves. Consequences: **Sky3D is IN** (CPC non-profit, not open source; the SKILL.md must say so and point a commercial user at the authors). **GEF stays OUT**, unchanged, since it fails buildability, not licensing. **Theo4Exp stays OUT for a reason that was never about licensing**: it is a server-side web platform with no source distribution at all, so there is nothing to clone, build, benchmark or cross-build, and a "skill" for it could only be a guide to filling in someone else's web form, unverifiable and liable to break on their next redesign.

- 2026-07-23: **the skill scope covers the whole nuclear-physics community, not only the user's own research lines** (user ruling, when the heavy-ion and EOS survey showed no interface to Lines A/E/F). Consequence: heavy-ion collisions (both the relativistic hydro/transport chain and low-energy TDHF/QMD) and the nuclear/dense-matter equation of state become legitimate rows, and "it does not connect to my work" is NOT a reason to drop a code. The existing hard rules are unchanged and do all the filtering: published paper, publicly clonable, buildable from source on the target platform, Codex adversarial pass, honest benchmark tier. Practical consequence for ordering: these rows serve users the platform does not have yet, so they rank below a row that is still empty but above deepening a row that already has a tier-1 skill. Note the asymmetry the survey found: the codes CLOSEST to the user's own field (low-energy heavy-ion: ImQMD, AMD, CoMD, IBUU, NRV) are the ones that mostly FAIL the publicly-clonable rule, while the cleanly open ones sit at relativistic energies, so serving the community here genuinely means building away from home.

- 2026-07-20: **commit and push after each completed stage** (user directive), not in one batch at the end. Scope every commit by path. This repo currently sees concurrent Claude sessions, so `git commit -a` risks sweeping another session's in-progress work into the wrong commit; stage explicit paths and check `git status` before and after.

- 2026-07-24: **a skill's pass criterion is REPRODUCING THE PUBLISHED WORK, not a from-source-build ceremony** (user ruling). The real test of whether a skill is good is whether it reproduces the specific results reported in the code's paper (a figure, a table, a number), which is exactly the fusion-paper benchmark's L2 (reproduce the author reference case) and L3 (unseen system vs published values). The from-source-certification / anti-spoof rebuild I added on Thermal-FIST and vHLLE (`VERIFY OK` vs `PASSED-NOT-CERTIFIED`) is at most a secondary build-integrity detail; it is NOT the axis on which a skill passes, so **the TODO item to retrofit it across the family is dropped**, and the certification language must not be presented as the pass gate. A skill's job in this system is to supply a reference case plus a documented tolerance, ideally anchored to a published result; the fusion-paper harness (`/Users/jinlei/Desktop/code/fusion-paper`, `harness/run_cell.py` + verifiers) is where "did the agent reproduce the published work" is actually measured. This is why the published-work audit belongs with the paper, not the platform verify scripts. Consequence: skills whose benchmark currently reproduces only the code's OWN test suite (e.g. Thermal-FIST's ctest, SkyNet's self-comparison) still need a published-result anchor before they enter the paper matrix, or they drop in the 12-13-code selection.

- 2026-07-24: **FUSION is a long-term maintained platform; skill-building is neither urgent nor one-shot** (user ruling). The fusion-paper coverage gate CLOSED 2026-07-23 (one code per subfield row), the platform already has 20 per-code skills, and the paper prunes to a final 12-13, so **a new skill is platform value, not paper-critical**, and building more mainly adds pruning and disclosure burden. The heavy-ion row is the concrete over-build: SMASH + GiBUU + Thermal-FIST + vHLLE are four skills for what the paper treats as ONE (proposed fifth) row. Consequence: STOP the skill-building treadmill until the paper ships; the critical path is now the fusion-paper (Phase B harness, the 2026-08-31 axis + cell-list freeze, L3 published-anchor pre-registration, then the Sep-Oct matrix). New per-code skills resume as ordinary long-term platform maintenance, a few at a time, after submission (or when a specific code is actually wanted), never as a race to cover the field.
