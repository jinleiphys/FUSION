# Accelerating an Iterative Eigensolver for Nuclear Structure Configuration Interaction Calculations on GPUs using OpenACC

> To accelerate the solution of large eigenvalue problems arising from many-body calculations in nuclear physics on distributed-memory parallel systems equipped with general-purpose Graphic Processing Units (GPUs), we modified a previously developed hybrid MPI/OpenMP implementation of an eigensolver written in FORTRAN 90 by using an OpenACC directives based programming model. Such an approach requires making minimal changes to the original code and enables a smooth migration of large-scale nuclear structure simulations from a distributed-memory many-core CPU system to a distributed GPU system. However, in order to make the OpenACC based eigensolver run efficiently on GPUs, we need to take into account the architectural differences between a many-core CPU and a GPU device. Consequently, the optimal way to insert OpenACC directives may be different from the original way of inserting OpenMP directives. We point out these differences in the implementation of sparse matrix-matrix multiplications (SpMM), which constitutes the main cost of the eigensolver, as well as other differences in the preconditioning step and dense linear algebra operations. We compare the performance of the OpenACC based implementation executed on multiple GPUs with the performance on distributed-memory many-core CPUs, and demonstrate significant speedup achieved on GPUs compared to the on-node performance of a many-core CPU. We also show that the overall performance improvement of the eigensolver on multiple GPUs is more modest due to the communication overhead among different MPI ranks.

## Key claim

The paper demonstrates that a previously developed MPI/OpenMP hybrid iterative LOBPCG eigensolver for nuclear structure calculations can be ported to GPU systems using OpenACC directives with minimal code changes, but achieving significant speedup requires careful adaptation to GPU architecture differences, specifically, loop fusion and atomic updates, rather than a naive replacement of OpenMP with OpenACC directives.

## Method

The authors modified the existing FORTRAN 90 implementation of the LOBPCG eigensolver in the MFDn (Many-body Fermion Dynamics for nuclei) code by inserting OpenACC directives, replacing BLAS/LAPACK calls with cuBLAS and cuSOLVER library calls, and using CUDA-aware MPI for communication between GPUs. The key computational kernel, sparse matrix-matrix multiplication (SpMM), was optimized by fusing the two outermost loops using the OpenACC `collapse(2)` clause and fusing the two innermost loops, with atomic updates to handle write conflicts, and by introducing an additional level of cache blocking within each nonzero block of the local Hamiltonian. The preconditioning step was parallelized by decorating the loop over diagonal tiles with OpenACC `parallel loop` and using `loop vector` directives within the iterative FOM solver for each tile.

## Key numbers

- The number of vectors (eigenpairs sought) `k` is typically of the order of 8 to 16.
- The dimension `n` of the sparse symmetric Hamiltonian matrix `H` can be several billion.
- Block sizes within the CSB_Coo sparse matrix format are limited to 32,000, with practical block sizes of 4,000 to 8,000.
- The number of vector threads in the innermost loop is typically 4, 8, or 16, which is significantly less than the minimum number of vector threads on current GPUs.

## Context

This work extends the hybrid MPI/OpenMP implementation of the LOBPCG eigensolver previously developed for many-core CPU systems such as the Cori KNL system at NERSC (Shao et al., 2018; Aktulga et al., 2014, 2016). The techniques developed are applicable to other block-iterative eigensolvers such as the Davidson method (Davidson, 1975) and the block-Lanczos algorithm. The paper enables nuclear structure simulations to run efficiently on distributed-memory multi-GPU systems, though the authors note that for large problems distributed among hundreds of GPUs, speedup factors are limited by data communication overhead among different GPUs.
