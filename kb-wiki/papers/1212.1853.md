# Generating random thermal momenta

> Generation of random thermal particle momenta is a basic task in many problems, such as microscopic studies of equilibrium and transport properties of systems, or the conversion of a fluid to particles. In heavy-ion physics, the (in)efficiency of the algorithm matters particularly in hybrid hydrodynamics + hadronic transport calculations. With popular software packages, such as UrQMD 3.3p1 or THERMINATOR, it can still take ten hours to generate particles for a single Pb+Pb "event" at the LHC from fluid dynamics output. Below I describe reasonably efficient simple algorithms using the MPC package, which should help speed momentum generation up by at least one order of magnitude. It is likely that this wheel has been reinvented many times instead of reuse, so there may very well exist older and/or better algorithms that I am not aware of (MPC has been around only since 2000). The main goal here is to encourage practitioners to use available efficient routines, and offer a few practical solutions.

# Generating random thermal momenta

## Key claim
The paper describes reasonably efficient algorithms for generating random thermal particle momenta using the MPC package that should speed momentum generation up by at least one order of magnitude compared to popular software packages such as UrQMD 3.3p1 or THERMINATOR, though the author notes that older or better algorithms may already exist and the main goal is to encourage reuse of available efficient routines.

## Method
The paper presents algorithms for generating random thermal momenta from static thermal distributions using rejection methods with staircase comparison functions or tabulated interpolation, then extends these to boosted thermal distributions with collective flow and Cooper-Frye freezeout across spacetime hypersurfaces. For Cooper-Frye freezeout, the method generates thermal momenta in the local rest frame of the fluid element, applies an additional rejection step to account for the momentum-dependent prefactor from the surface normal, and boosts back to the original frame. A specialized one-liner method is given for massless Boltzmann particles using three independent exponential deviates.

## Key numbers
- With popular software packages such as UrQMD 3.3p1 or THERMINATOR, it can take ten hours to generate particles for a single Pb+Pb event at the LHC from fluid dynamics output.
- For static thermal distributions using the interpolation method, the efficiency benchmark epsilon = 3/(uniform deviates per momentum) equals 1.
- For boosted thermal distributions using the interpolation method, epsilon is at least 3/8.
- For Cooper-Frye freezeout using the interpolation method, epsilon is at least 3/16.
- With automated rejection, the formal efficiency drops to epsilon at least 3/4 for static, at least 3/10 for boosted, and at least 3/20 for Cooper-Frye.
- For timelike normals in Cooper-Frye freezeout, no more than two tries are needed on average; for spacelike normals with n^0 >= 0, no more than four tries.
- For spacelike normals with n^0 >= 0, the efficiency is at least |v|/4 for the cut prefactor method, and at least 1/4 for the direct generation method using modified comparison function.
- In practice, overall efficiency is a reasonable approximately 0.1 even with the automated rejection method.

## Context
This work addresses the practical computational bottleneck in hybrid hydrodynamics plus hadronic transport calculations for heavy-ion physics, where generating thermal momenta from fluid dynamics output can be extremely slow with existing packages like UrQMD 3.3p1 or THERMINATOR. The paper extends standard rejection method techniques (described in standard references) and Cooper-Frye freezeout (Cooper and Frye, 1974) to provide efficient algorithms implemented in the MPC package, which has been available since 2000. The work enables faster conversion of fluid dynamics output to particles in heavy-ion event simulations, particularly for LHC Pb+Pb collisions, and encourages practitioners to use available efficient routines rather than reinventing less efficient implementations.
