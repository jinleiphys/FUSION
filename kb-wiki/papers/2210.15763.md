# Bootstrapped Block Lanczos for large-dimension eigenvalue problems

> The Lanczos algorithm has proven itself to be a valuable matrix eigensolver for problems with large dimensions, up to hundreds of millions or even tens of billions. The computational cost of using any Lanczos algorithm is dominated by the number of sparse matrix-vector multiplications until suitable convergence is reached. Block Lanczos replaces sparse matrix-vector multiplication with sparse matrix-matrix multiplication, which is more efficient, but for a randomly chosen starting block (or pivot), more multiplications are required to reach convergence. We find that a bootstrapped pivot block, that is, an initial block constructed from approximate eigenvectors computed in a truncated space, leads to a dramatically reduced number of multiplications, significantly outperforming both standard vector Lanczos and block Lanczos with a random pivot. A key condition for speed-up is that the pivot block have a non-trivial overlap with the final converged vectors. We implement this approach in a configuration-interaction code for nuclear structure, and find a reduction in time-to-solution by a factor of two or more, up to a factor of ten.

## Key claim

The authors find that using a bootstrapped pivot block (constructed from approximate eigenvectors computed in a truncated model space) dramatically reduces the number of matrix multiplications needed for convergence in block Lanczos, but only if the pivot block has a non-trivial overlap with the final converged vectors; otherwise intruder states orthogonal to the target eigenvectors can slow convergence.

## Method

The authors implemented block Lanczos in the configuration-interaction code BIGSTICK, which uses on-the-fly reconstruction of Hamiltonian matrix elements rather than storing them. They constructed bootstrapped pivot blocks by computing approximate eigenvectors in smaller truncated model spaces and projecting them into the full working space. They tested the method on two nuclear systems: a full-configuration calculation of 53Mn in the pf-shell (141 million basis dimension) and a no-core shell model calculation of 7Be with Nmax=8 truncation (6.1 million basis dimension), comparing performance against standard vector Lanczos and block Lanczos with random pivots.

## Key numbers

- For 53Mn full-configuration calculation (141 million basis dimension): bootstrapped block Lanczos with block size 12 and W=6 truncation pivot took 1.28 x 10^3 seconds, compared to 12 x 10^3 seconds for single-vector Lanczos and 14 x 10^3 seconds for conventional block Lanczos with random pivot (approximately 9x speedup over single-vector Lanczos)
- For 53Mn: single-vector Lanczos required 360 vector-equivalent iterations to converge 12 states, while bootstrapped block Lanczos with W=6 truncation required only 84 vector-equivalent iterations
- For 53Mn: time per operation decreased from 3.3 ns for block size 1 (single-vector) to 1.2 ns for block size 12, a factor of approximately 2.75 improvement
- For 53Mn: the three truncated model spaces used for bootstrapping had dimensions of 1.65 million (W=4), 7.95 million (W=5), and 25.5 million (W=6)
- For 7Be: the three truncated model spaces used for bootstrapping had dimensions of 1961 (Nmax=2), 49 thousand (Nmax=4), and 660 thousand (Nmax=6)

## Context

This work extends the block Lanczos approach previously applied in nuclear structure by Shimizu (2019), who demonstrated that careful construction of the initial pivot block using approximate eigenvectors could accelerate convergence in restarted block Lanczos. The authors implement this bootstrapping idea in non-restarted block Lanczos within the BIGSTICK code, finding that even modest truncations can yield significant speedups, though they caution that intruder states (states orthogonal to the target eigenvectors due to level crossings as the model space increases) can negate the benefits. The method enables faster computation of low-lying nuclear spectra for very large configuration-interaction problems where time-to-solution on supercomputers is a major concern.
