# The JETSCAPE framework

> The JETSCAPE simulation framework is an overarching computational envelope for developing complete event generators for heavy-ion collisions. It allows for modular incorporation of a wide variety of existing and future software that simulates different aspects of a heavy-ion collision. The default JETSCAPE package contains both the framework, and an entire set of indigenous and third party routines that can be used to directly compare with experimental data. In this article, we outline the algorithmic design of the JETSCAPE framework, define the interfaces and describe the default modules required to carry out full simulations of heavy-ion collisions within this package. We begin with a description of the various physics elements required to simulate an entire event in a heavy-ion collision, and distribute these within a flowchart representing the event generator and statistical routines for comparison with data. This is followed by a description of the abstract class structure, with associated members and functions required for this flowchart to work. We then define the interface that will be required for external users of JETSCAPE to incorporate their code within this framework and to modify existing elements within the default distribution. We conclude with a discussion of some of the physics output for both $p$-$p$ and $A$-$A$ collisions from the default distribution, and an outlook towards future releases. In the appendix, we discuss various architectures on which this code can be run and outline our benchmarks on similar hardware.

## Key claim

The JETSCAPE framework provides a modular, task-based C++ event generator for heavy-ion collisions that allows users to incorporate their own physics modules by overloading base classes, but the default distribution's physics output is presented only as a demonstration and the framework's performance benchmarks are limited to specific hardware configurations (Wayne State University Grid nodes and Stampede login nodes).

## Method

The framework implements an abstract class structure where each factorizable physics component (initial state, hydrodynamics, energy loss, hadronization) is set up as a separate module that inherits from base classes. Modules are connected and executed by a task-based system that initializes and executes them recursively, conveying requisite information between modules via shared data structures including a parton shower graph representation. The default distribution includes indigenous and third-party routines such as TRENTO for initial state, MUSIC for hydrodynamics, and MATTER, LBT, MARTINI, and AdS/CFT for energy loss, which are attached to a main JetScape task via managers.

## Key numbers

* Proton-proton collisions at 2.76 TeV: total runtime of 150 seconds for 1,000 events, corresponding to approximately 0.15 seconds per event, with memory consumption of approximately 150 MB on a Wayne State University Grid node with Intel E5-2643 processor (3.3 GHz clock speed).
* For proton-proton collisions, the MATTER energy loss module dominates runtime at 125 seconds total (0.125 seconds per event), while TRENTO initial state takes 3 seconds (0.003 seconds per event) and BrickHydro takes 0.04 seconds (4x10^(-5) seconds per event).
* For heavy-ion collisions (single event), the MUSIC hydrodynamics module requires 341 MB of memory and 112 seconds of runtime, while the freestreaming module requires 71 MB and 1 second.
* GPU optimization using AMD GPUs can yield a factor of 6 increase in performance compared to multi-core CPUs for 3+1D hydrodynamics modules, as found in work by Pang (2018).

## Context

This paper describes the JETSCAPE framework version 1.3, which extends the ecosystem of heavy-ion collision event generators by providing a modular platform that can incorporate existing codes such as TRENTO, MUSIC, MATTER, LBT, MARTINI, and AdS/CFT energy loss formalisms. The framework enables systematic comparison of different physics models within a unified computational envelope and facilitates the development of new modules by external users. Future work includes optimizing the most computation-intensive parts (particularly hydrodynamics) using GPU acceleration, as demonstrated by Pang (2018) who found a factor of 6 speedup with AMD GPUs.
