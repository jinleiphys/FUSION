# MAGE-HEP: Monte Carlo Analysis and Graphical Environment for High-Energy Physics

> Monte Carlo event generators are central to high-energy physics analysis. However, workflows based on handwritten scripts can be difficult to reuse, modify, and reproduce when multiple Monte Carlo models, tune variations, run variations, and output formats are involved. We present MAGE-HEP, short for Monte Carlo Analysis and Graphical Environment for High-Energy Physics, a Graphical User Interface (GUI) driven workflow environment for reproducible Monte Carlo-based analyses in high-energy physics. MAGE-HEP organizes analysis workflows through a project-study-run hierarchy. The project stores the workspace, the study stores the reusable analysis context, and each run represents a controlled execution of that context. The MAGE-HEP Node API provides the analysis-building layer for defining generator configurations, observables, selections, output rules, and generated C++/ROOT analysis code. A study context can be inspected, reused, or exported as a \texttt{.mcx} context bundle, while the project state can be exported as a portable \texttt{.mgp} bundle. The current beta implementation validates the core idea using a PYTHIA8 and ROOT workflow. It includes background execution, manifest-based run tracking, live ROOT inspection, and particle-table summaries for supported output layouts. This paper describes the architecture, workflow, and current beta implementation of MAGE-HEP.

## Key claim

MAGE-HEP introduces a project-study-run hierarchy and a reusable study context (exportable as `.mcx` bundles) to address reproducibility in Monte Carlo generator-level analyses, but the current beta implementation validates this idea only for a PYTHIA8 and ROOT workflow, with cross-generator support treated as a future design direction rather than a completed feature.

## Method

The authors developed a Qt/C++ graphical user interface (GUI) that organizes Monte Carlo analyses through a three-level hierarchy: project (workspace and metadata), study (reusable analysis context including generator configuration, observable definitions, and output rules), and run (a controlled execution with specific seed, parameters, and output). The MAGE-HEP Node API allows users to define analysis contexts in C++ using predefined recipes and user-defined observables; these contexts are then transpiled into generated C++/ROOT analysis code. Runs are executed in the background via a `mage-daemon` module using separate `tmux` sessions, with manifest-based run tracking, live ROOT inspection, and particle-table summaries for supported output layouts.

## Key numbers

* The beta implementation validates the core idea using a PYTHIA8 and ROOT workflow with a center-of-mass energy of 13.6 TeV for proton-proton collisions and 5.36 TeV per nucleon pair for lead-lead collisions.
* The example user-defined context specifies 100,000,000 events for the proton-proton source and 10,000 events for the heavy-ion source.
* The predefined identified-particle spectra recipe uses a pseudorapidity acceptance cut of `etaMax = 3.0`, a global rapidity cut of `yMaxGlobal = 0.8`, a transverse momentum range of 0.0 to 10.0 GeV/c with bin width 0.1 GeV/c, and a minimum transverse momentum of 0.05 GeV/c.
* The azimuthal-angle analysis example uses a binning of 0.0 to 6.28318 radians with bin width 0.0628318 radians, and a pseudorapidity cut of `etaMax = 0.8`.

## Context

This work extends the ecosystem of high-energy physics analysis tools by providing a context-driven workflow layer on top of existing Monte Carlo generators and analysis frameworks. It contrasts with tools like Rivet (Bierlich 2019) for particle-level analysis preservation, Professor (Buckley 2009) for systematic event-generator tuning, and MadAnalysis 5 (Conte 2012) for collider phenomenology, which do not define the generator-level analysis logic itself as a reusable object. The framework enables early-stage users to build reproducible generator-level workflows and allows experienced users to manage repeated analyses, parameter variations, and output metadata without manually maintaining separate scripts; future versions could integrate with REANA (reana) for larger analysis preservation environments and extend to additional generator backends such as HepMC-based event records.
